# P0 / P1 Production Hardening — Change Log

This document records every security and correctness fix landed during the
pre-release hardening pass. Changes are grouped by priority tier. File
references are relative to the repo root.

## Tier 0 — Incident Fixes (blocking, found during smoke tests)

These were surfaced by console errors in the browser while testing the
Superadmin and login flows. They read like CORS/React bugs but all three
had unrelated root causes.

### 0.1  `POST /api/v1/auth/login → 422 Unprocessable Entity`
- **Cause:** `backend/.env` contained `SUPERADMIN_EMAIL=admin`, which isn't a
  valid email. `LoginRequest.email: EmailStr` rejects it with a Pydantic
  422 whose `detail` is a list of validation-error objects.
- **Fix:** `backend/.env` → `SUPERADMIN_EMAIL=admin@altrium.com`.
- **Side effect uncovered:** the superadmin seeder in `main.py`'s lifespan
  had been silently failing on every boot (the `UserCreate(...)` call
  raised `ValidationError` on `admin`, caught by a bare `except`). No
  superadmin had ever been created. The new value lets the seeder run.

### 0.2  `Minified React error #31` on login failure
- **Cause:** FastAPI 422 bodies have `detail` as an array of
  `{type, loc, msg, input, ctx}` dicts. The frontend's
  `toast.error(detail || "…")` passed the raw array to React, which
  can't render objects as children.
- **Fix:** added `frontend/src/utils/errors.ts` (`extractErrorMessage`)
  that coerces any FastAPI 422/4xx payload into a string, preserving
  field context (`field: message; …`). Wired into:
  - `frontend/src/pages/Login.tsx`
  - `frontend/src/pages/SuperadminDashboard.tsx`
  - `frontend/src/pages/StudentDashboard.tsx`
  - `frontend/src/pages/EmployerVerify.tsx`

### 0.3  `GET /api/v1/users/me → 500 + "No 'Access-Control-Allow-Origin'"`
- **Cause (primary):** stale `access_token` in `sessionStorage` contained a
  `sub` that wasn't a valid UUID. `backend/app/api/deps/auth.py` did
  `UUID(user_id)` unwrapped. `ValueError` → caught by the generic
  `@app.exception_handler(Exception)` → 500 returned via Starlette's
  `ServerErrorMiddleware`, which is outside the CORS middleware, so the
  response had no `Access-Control-Allow-Origin` header.
- **Fix:**
  - `backend/app/api/deps/auth.py` — wrap `UUID(user_id)` in try/except,
    raise `HTTPException(401, "Invalid token")`.
  - `backend/app/core/errors.py` — 500 handler now attaches CORS headers
    manually (`Access-Control-Allow-Origin`, `Access-Control-Allow-Credentials`,
    `Vary: Origin`) so unexpected server errors no longer masquerade as
    CORS failures.

### 0.4  `GET /api/v1/users/ → 500`  (found on Superadmin dashboard)
- **Cause:** one legacy user in Mongo had `email="admin"` (from the pre-fix
  seed) and three admins had `wallet_address=""`. `UserResponse` inherited
  `EmailStr` and the `WALLET_ADDRESS_PATTERN`, so `ResponseValidationError`
  blew up on every list call.
- **Fix:** `UserResponse` no longer inherits `UserBase`. `email` is plain
  `str`; `wallet_address` is unconstrained at the response level. Input
  validation (`UserCreate`, `RegisterRequest`) is unchanged.
- **Follow-up for ops:** optionally clean up legacy rows with
  `db.users.deleteOne({ email: "admin" })`.

### 0.5  Dialog a11y warning
- Superadmin "Students per college" dialog had `DialogTitle` but no
  `DialogDescription`. Added a description so Radix stops warning.
- `frontend/src/pages/SuperadminDashboard.tsx`.

## Tier 1 — P0 (must fix before production)

### P0.1  `/degrees/*` admin-verification regression
- `credentials.py` correctly used `require_verified_admin`, but the newer
  `degrees.py` router used only `require_role(UserRole.ADMIN)`. Unverified
  admins could approve, revoke, update, delete, and reset degrees through
  `/degrees/*`, bypassing the legal-stakeholder gate entirely.
- **Fix:** every mutation route in `backend/app/api/routes/degrees.py`
  switched to `require_verified_admin`, then in P0.6 tightened further
  to `require_admin_with_wallet`.

### P0.2  Blacklisted tokens never auto-expired
- Beanie's `indexes = ["expires_at"]` creates a plain index, not a TTL
  index. Blacklisted tokens accumulated forever.
- **Fix:** `backend/app/models/models.py` →
  ```python
  indexes = [
      IndexModel([("token", ASCENDING)], unique=True),
      IndexModel([("expires_at", ASCENDING)], expireAfterSeconds=0),
  ]
  ```
  Mongo now purges expired tokens automatically; `token` is uniquely
  indexed so duplicate inserts are no-ops.

### P0.3  Access / refresh tokens were interchangeable
- Both token types were signed with the same key and validated by the same
  `decode_token` with no `typ` claim. A stolen access token could be used
  as a refresh token (and vice versa).
- **Fix:** `backend/app/core/security.py`:
  - `create_access_token` adds `"type": "access"`.
  - `create_refresh_token` adds `"type": "refresh"`.
  - `decode_token(token, expected_type=...)` rejects mismatches.
  - New `get_token_exp(token)` returns the real `exp` for blacklist sizing.
- Consumers updated: `deps/auth.py` (requires access), `services/auth_service.py`
  (refresh requires refresh).

### P0.4  Logout didn't revoke refresh tokens; blacklist used wrong TTL
- Old `/auth/logout`:
  - Only blacklisted the access token.
  - Used `now + ACCESS_TOKEN_EXPIRE_MINUTES` as `expires_at`, ignoring the
    token's real `exp`.
- **Fix:** `backend/app/api/routes/auth.py`:
  - Accepts an optional `{"refresh_token": "..."}` body.
  - Blacklists both access and refresh tokens using their actual JWT `exp`.
  - `AuthService.refresh` now checks the blacklist before issuing a new
    pair, so a stolen refresh token can't outlive logout.
  - Frontend `AuthContext.logout` sends the refresh token with the logout
    request.

### P0.5  PDF upload validation was weak
- Before: only trusted client-supplied `Content-Type`; `pypdf` parsed the
  first 1 MiB only; no size cap.
- **Fix:** new shared module `backend/app/services/pdf_validation.py`
  (`validate_pdf_upload`) enforces, in order:
  1. `Content-Type == application/pdf`.
  2. Streamed 10 MiB hard size limit (cannot OOM the server).
  3. Magic-byte check (payload must start with `%PDF-`).
  4. Full-file `pypdf.PdfReader` parse.
  Returns the validated bytes so callers don't re-read.
- Applied in:
  - `backend/app/services/degree_service.py::DegreeService.upload_document`
  - `backend/app/api/routes/auth.py::upload_verification_document`
    (previously had **no** PDF validation at all).

### P0.6  On-chain minting allowed without `UNIVERSITY_ROLE` at API level
- `is_legal_admin_verified=True` at the DB level doesn't imply the admin
  holds the on-chain `UNIVERSITY_ROLE` — that only happens after they
  connect a wallet via `PATCH /users/me/wallet`. API endpoints that
  produce on-chain state would accept the request and fail on-chain,
  wasting gas and giving a poor error.
- **Fix:** new dependency `require_admin_with_wallet` in
  `backend/app/api/deps/auth.py`: layered on top of `require_verified_admin`
  and additionally requires `current_user.wallet_address`. `SUPERADMIN`
  is exempt (they don't touch chain state directly).
- Applied to every on-chain-adjacent endpoint:
  - `PATCH /credentials/{id}/status`
  - `PATCH /credentials/{id}`
  - `DELETE /credentials/{id}`
  - `PATCH /degrees/{id}/status`
  - `PATCH /degrees/{id}`
  - `PATCH /degrees/{id}/revoke`
  - `POST /degrees/{id}/reset`
  - `DELETE /degrees/{id}`
- Read-only listings (`GET /users/`, `GET /users/my-students`) remain on
  `require_verified_admin` — dashboards work without a wallet.

### P0.7  Wallet PATCH gated on verification
- Admins must not be able to connect a wallet before the Superadmin has
  approved them (per the required sequence: approve → connect → role grant).
- **Fix:** `PATCH /users/me/wallet` switched from
  `require_role(ADMIN, SUPERADMIN)` → `require_verified_admin`.
  Unverified admins now receive 403 "University admin account pending legal
  verification by Superadmin" from this endpoint too.

### P0.8  Fail loud on insecure production config
- `ENVIRONMENT=production` with any of the known-insecure defaults would
  previously boot fine.
- **Fix:** `backend/app/main.py::_enforce_production_security` raises
  `RuntimeError` at startup in production if any of:
  - `SECRET_KEY` is empty, `< 32` chars, or `"change-me-in-production"` /
    `"your-secret-key-change-in-production"`.
  - `SUPERADMIN_PASSWORD` is empty, `< 12` chars, or one of
    `{"123", "admin", "password", "changeme"}`.
  - `BACKEND_CORS_ORIGINS` contains `"*"`.
  - `ALLOWED_HOSTS` contains `"*"`.
- `TrustedHostMiddleware` now reads from `settings.ALLOWED_HOSTS`
  (previously hardcoded `["*"]`).
- `.env` grew `ALLOWED_HOSTS=["localhost","127.0.0.1","backend"]` for dev.

### P0 — Final admin onboarding flow (for reference)

| Step | Endpoint | Gate | Effect |
|------|----------|------|--------|
| 1 | `POST /auth/register` | public | Admin row created, `is_legal_admin_verified=False`, no wallet |
| 2 | — | — | Admin logs in; pending-verification screen only. All `require_verified_admin` routes 403, including `/me/wallet` |
| 3 | `POST /users/verify-admin/{id}` | SUPERADMIN | Flips DB flag; admin is now "valid_admin" |
| 4 | dashboard reads | `require_verified_admin` | Valid_admin sees `/users/my-students` + dashboard; mutations still 403 via `require_admin_with_wallet` |
| 5 | `PATCH /users/me/wallet` | `require_verified_admin` | Saves EVM address; since both conditions now met, `addUniversity(wallet)` fires on-chain and grants `UNIVERSITY_ROLE` |
| 6 | mint / revoke / reset | `require_admin_with_wallet` | API gate passes; on-chain tx will also succeed because role is held |

If step 5 never runs, step 3's DB flag is effectively inert for anything
on-chain — the invariant requested by product.

## Tier 2 — P1 (pre-release hardening)

### P1.1  Password policy
- **Before:** `password: str` — accepted any non-empty string.
- **After** (`backend/app/schemas/schemas.py::UserCreate`):
  - `Field(..., min_length=12, max_length=128)`
  - `field_validator` requires at least one lower, one upper, one digit,
    and one symbol.
- Tests and `.env` superadmin seed updated to comply
  (`AltriumDev!2024Admin`). Existing superadmin row keeps its old hash
  because the seeder is `if not existing`.
- Frontend register form (`frontend/src/pages/Register.tsx`) mirrors the
  constraint with `minLength`, `maxLength`, a `pattern`, and a visible
  hint below the field.

### P1.2  Rate limiting coverage

| Endpoint | Before | After |
|----------|-------:|------:|
| `POST /auth/login` | 50/min | **5/min** |
| `POST /auth/register` | 50/min | **10/min** |
| `POST /auth/refresh` | 50/min | **30/min** |
| `POST /auth/{id}/verification-document` | — | **20/min** |
| `POST /degrees/{id}/document` | — | **20/min** |
| `GET /degrees/public` | — | **30/min** |
| `GET /credentials/public` | — | **30/min** |

Login tightened from a cosmetic 50/min (3 000/hour — useless against
credential stuffing) to 5/min per IP. Uploads and public scrape surfaces
are now bounded as well.

### P1.3  Security response headers — `SecurityHeadersMiddleware`
New module `backend/app/core/security_headers.py`. Applied in `main.py`
ahead of gzip/trusted-host. Defaults:

- `X-Content-Type-Options: nosniff`
- `X-Frame-Options: DENY`
- `Referrer-Policy: strict-origin-when-cross-origin`
- `Permissions-Policy: geolocation=(), microphone=(), camera=(), payment=()`
- `Cross-Origin-Opener-Policy: same-origin`
- `Cross-Origin-Resource-Policy: same-site`
- `Content-Security-Policy: default-src 'none'; frame-ancestors 'none'`
  (API returns JSON/PDF only — no inline execution surface)
- `Strict-Transport-Security: max-age=63072000; includeSubDomains; preload`
  (production environment only)

### P1.4  Regex / input-format audit

Gaps closed:

| Field | Before | After |
|-------|--------|-------|
| `wallet_address` on register | `^0x[a-fA-F0-9]{40}$` ✓ | unchanged |
| `wallet_address` on `PATCH /me/wallet` | **bypassed** — raw `dict` | new `WalletPatchRequest` enforces `^0x[a-fA-F0-9]{40}$`; value re-checksummed to EIP-55 before save |
| `tx_hash` | `^0x[a-fA-F0-9]{64}$` ✓ | unchanged |
| `prn_number` | unvalidated | `^[A-Za-z0-9\-]{3,32}$` (feeds on-chain `keccak(prn-college)`) |
| `full_name` | unvalidated | `max_length=100` + control-char strip |
| `college_name` | unvalidated | `max_length=150` + control-char strip |
| `title` | unvalidated | `min_length=1, max_length=200` + strip |
| `description` | unvalidated | `max_length=2000` + strip |
| `token_id` | unvalidated | `ge=0` |

Non-EVM wallets (Solana Base58, Bitcoin Base58/Bech32, Cosmos Bech32,
etc.) fail the Ethereum-address regex by construction. Because
`PATCH /users/me/wallet` now routes through `WalletPatchRequest`, a
non-Ethereum wallet produces a 422 before the DB is touched. No wallet
save → no `UNIVERSITY_ROLE` grant → `require_admin_with_wallet` keeps
blocking every web3 action. The only way an admin reaches chain
interaction is via MetaMask (or any other EVM wallet) connected with a
valid 20-byte hex address.

## Migration / rollout notes

1. **Rebuild required.** Changes are baked into container images; restart
   alone isn't enough:
   ```
   docker compose up -d --build backend
   docker compose up -d --build frontend
   ```
2. **Existing superadmin row** — the pre-policy row with hash of `"123"`
   was bulk-reset in T3.4 to `Altrium123!Dev`. Login works immediately.
   The policy-compliant seed also takes effect on a fresh DB.
3. **Legacy DB rows** with `email="admin"` or `wallet_address=""` no longer
   break `/users/`. The `email="admin"` superadmin was deleted in T3.4.
   Any remaining `wallet_address=""` rows can optionally be cleaned up:
   ```
   db.users.updateMany({ wallet_address: "" }, { $set: { wallet_address: null } })
   ```
4. **Existing admins without a wallet** will start getting 403s on mint
   endpoints after rebuild. That's intentional — they must hit
   `PATCH /users/me/wallet` with a MetaMask (EVM) address.
5. **Old blacklisted-token rows** (from before the TTL index) will be
   dropped by the TTL as soon as their `expires_at` passes. No manual
   migration is needed unless you want them gone immediately.
6. **Production boot** now refuses to start with `SECRET_KEY=change-me-…`,
   `SUPERADMIN_PASSWORD=123`, wildcard CORS, or wildcard allowed-hosts.
   Make sure the production env vars are set before flipping
   `ENVIRONMENT=production`.

## Tier 3 — Post-release hotfixes, UX upgrades, and legacy migration

Landed immediately after the P1 pass while running the first full end-to-end
walk-through. Everything here is live in the running containers.

### T3.1  Blacklisted-token TTL index conflict (hotfix — backend crash-loop)
- **Symptom:** After rebuild, backend container restart-looped; browser saw
  `ERR_CONNECTION_REFUSED` on every request.
- **Cause:** The pre-P0 model had `indexes = ["expires_at"]` (plain index
  named `expires_at_1`). The P0 patch redefined that index as TTL
  (`expireAfterSeconds=0`) under the same auto-generated name. Mongo
  refuses to redefine an index with different options →
  `pymongo.errors.OperationFailure` code 85 (`IndexOptionsConflict`) →
  `init_beanie` raised → lifespan startup failed → uvicorn exited.
- **Fix:** `backend/app/main.py::_reconcile_blacklist_indexes` runs on
  every boot **before** `init_beanie`. It inspects
  `blacklisted_tokens.index_information()` and drops any `expires_at`
  index that is **not** already a TTL index. Beanie then recreates the
  correct TTL index cleanly. Idempotent: once the TTL index exists,
  subsequent boots find it and do nothing.

### T3.2  In-app forgot-password flow (no email)
New self-serve reset so legacy accounts without real email addresses can
migrate into the new password policy without setting up SMTP.

- **Schemas (`schemas.py`):** new `_PasswordPayload` base class carries
  the same complexity validator as `UserCreate`. Subclasses:
  - `ForgotPasswordRequest { email: EmailStr, new_password: str }`
  - `ChangePasswordRequest { old_password: str, new_password: str }`
- **Service (`auth_service.py`):**
  - `reset_password_direct(email, new_password) -> bool` looks the user
    up and overwrites the hash; returns whether a user was found.
  - `change_password(user, old, new)` verifies the old password, rejects
    no-op rotations, then overwrites.
- **Routes (`auth.py`):**
  - `POST /auth/forgot-password` — rate-limited 5/min. Dev-mode only
    (gated by `settings.ALLOW_SELF_SERVE_PASSWORD_RESET`). Returns 404
    with `"No account found with that email"` if the email doesn't
    match, so admins get real feedback instead of a silent 204.
  - `POST /auth/change-password` — authenticated, rate-limited 10/min.
- **Config (`config.py`):** `ALLOW_SELF_SERVE_PASSWORD_RESET: bool = True`
  (dev). The production-config guard in `main.py` refuses to boot if
  this is true in production, forcing the eventual email-token flow.
- **Frontend:**
  - `src/components/PasswordStrengthChecklist.tsx` — live checklist with
    five green-tick/grey-circle rules + a colour-graded strength bar
    (red → orange → yellow → green). Also exports `evaluatePassword`
    and `isPasswordValid` for programmatic use.
  - `src/pages/ForgotPassword.tsx` — new route `/forgot-password` with
    email + new password + confirm + live checklist + match indicator.
  - `src/pages/Register.tsx` — static hint replaced with the live
    checklist.
  - `src/pages/Login.tsx` — "Forgot your password?" link above the
    register link.
  - `src/App.tsx` — route wired.

### T3.3  Case-insensitive email lookup
- **Symptom:** During T3.2 rollout, forgot-password silently reported
  success while users still got 401 on login — email case mismatch
  between DB and form input.
- **Fix:** `UserCRUD._norm_email(email) -> email.strip().lower()` applied
  in `create`, `get_by_email`, and downstream. Existing rows are only
  lowercased as they're next touched; no destructive migration required.
- **Honest reset response:** `reset_password_direct` now returns `bool`,
  and the dev endpoint 404s on miss so admins see the real outcome.

pass. None are P0.

  wired in T3** — we're running with non-deliverable test-domain emails

  arrival. To be revisited once real SMTP + real user accounts exist.

  `"Email already registered"` on duplicate — discloses existence).
- JWT storage strategy (`sessionStorage` → `HttpOnly` cookie + CSRF, or
  strict CSP for the SPA origin).
- Randomise upload filenames (defence-in-depth against IDOR if `uploads/`
  is ever served statically).
- Narrow generic `Exception` handler in `core/errors.py` into specific
  handlers (`RequestValidationError`, `ResponseValidationError`, etc.)
  so 500s can route through CORS middleware naturally instead of needing
  the manual header patch.
- Log-review pass for accidental PII / token emission.
- Replace `ALLOW_SELF_SERVE_PASSWORD_RESET` with a proper email-token
  reset flow before enabling production (`ENVIRONMENT=production`).
